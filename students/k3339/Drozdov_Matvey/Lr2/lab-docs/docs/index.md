# Документация: Сравнение подходов к параллелизму и конкурентности в Python

## 1. Описание задачи

Целью данной работы является реализация и сравнение трех подходов к выполнению задач в Python: **многопроцессорности (multiprocessing)**, **многопоточности (threading)** и **асинхронности (asyncio)**.

---

## 2. Описание программ

### 2.1 Задача суммирования

#### Многопроцессорность (`multiprocessing_sum.py`)

Этот скрипт реализует параллелизм на уровне операционной системы. 

##### Как это работает:
* Модуль `multiprocessing` создает пул независимых рабочих процессов (по умолчанию равный количеству ядер процессора или переданному значению `chunks`).
* Каждый процесс получает свою собственную копию интерпретатора Python и собственное выделенное пространство в оперативной памяти.
* Благодаря этому процессы полностью обходят GIL и выполняются на разных ядрах процессора абсолютно одновременно.
* Функция `pool.starmap` распределяет диапазоны чисел между процессами и собирает результаты в единый список.

---

#### 2. Многопоточность (`threading_sum.py`)

Этот скрипт реализует суммирование на основе потоков операционной системы в рамках одного процесса.

##### Как это работает:
* Модуль `threading` создает несколько потоков, которые делят между собой общую оперативную память программы.
* Для сбора результатов используется общий список `results`, куда каждый поток записывает вычисленную сумму строго по своему индексу (`index`). 
* Из-за особенностей эталонной реализации Python, работу потоков контролирует GIL. 


---

#### 3. Асинхронность (`async_sum.py`)

Этот скрипт использует **кооперативную многозадачность** на базе цикла событий (Event Loop) в рамках одного потока и одного процесса.

##### Как это работает:
* Модуль `asyncio` позволяет запускать задачи одновременно с помощью функции `asyncio.gather`.
* Функция вычисления `calculate_sum_async` помечена ключевым словом `async def`, что делает её корутиной.
* Однако внутри используется стандартная синхронная функция `sum(range(...))`. Поскольку она не содержит ключевого слова `await`, она становится **блокирующей**.

---

### 2.2 Задача парсинга (I/O-bound)

#### Асинхронность (`async_parser.py`)
Реализует полностью неблокирующий подход к сетевым запросам и работе с базой данных.
##### Как это работает:
* Для сетевых запросов используется асинхронная библиотека `aiohttp`, а для базы данных — драйвер `asyncpg` с пулом соединений (`create_pool`).
* Внедрен инструмент `asyncio.Semaphore(5)`, который ограничивает количество одновременных запросов к серверу до 5. 
* Цикл событий не простаивает: пока один запрос ждет ответа от сервера или БД, управление передается другой корутине.

#### Многопроцессорность (`multiprocess_parser.py`)
##### Как это работает:
* Создается пул процессов (`multiprocessing.Pool`), каждый из которых берет на себя один ID новости.
* Внутри процессов используется стандартная синхронная библиотека `requests` и синхронное подключение к БД (через `psycopg2`).
* Полная изоляция: ожидание ответа от сервера в одном процессе никак не влияет на работу остальных.

#### Многопоточность (`threading_parser.py`)
##### Как это работает:
* Создаются объекты `threading.Thread` для каждого URL. 
* В отличие от математической задачи, здесь GIL не мешает. Когда синхронная функция `requests.get()` ждет ответа по сети, поток засыпает и "отпускает" GIL, позволяя другим потокам в это время отправить свои сетевые запросы.

## 4. Анализ результатов и выводы

Сравнение времени выполнения:

1. В математической задаче лучше справился multiprocessing из-за разбиения на 4 ядра
2. В задаче парсинга разрыв между технологиями оказался минимальным. Все три подхода успешно справились с задачей распараллеливания сетевых запросов. 
   * **Threading** и **Multiprocessing** показали высокую скорость, так как во время ожидания ответа от сети (I/O sleep) процессор переключался на другие задачи. 
   * **Asyncio** в данном конкретном тесте показал результат чуть медленнее остальных. Это связано с так называемой "проблемой малых дистанций": на парсинг всего 5 ссылок программа тратит больше времени на поднятие архитектуры (инициализацию Event Loop, создание сессии `aiohttp` и установку пула соединений `asyncpg`), чем на сами запросы.
